steps_to_reproduce.non_executable.failure_types.missing_information.description:

steps_to_reproduce.non_executable.failure_types.ambiguous_information.description:

2. FIRST-PASS S2R DECISION
============================================================

Classify steps_to_reproduce.label as "Executable" when all of the following are true:
1. The Target Improved Report provides a concrete step cluster or concrete action path.
2. A tester can reasonably attempt the same intended bug from the given information.
3. The path requires only general Minecraft/game knowledge, not invented central setup.
4. The setup, action, and stated behavior point to one specific intended outcome.
5. Any omitted information is minor/common knowledge rather than a blocking precondition.

Classify steps_to_reproduce.label as "Non-Executable" when at least one blocking issue exists:
- a critical action, parameter, precondition, state, item, mob, block, command argument, environment, layout, scale, setting, or sequence is absent;
- the tester must invent the central trigger or perform extra investigation before attempting reproduction;
- a wrong command/action/item contradicts Minecraft mechanics and blocks execution;
- the steps are too ambiguous and allow multiple materially different reproduction paths;
- the report only describes the symptom without a concrete path to attempt it.

Important balance rule from YAML:
Omissions of minor steps of common knowledge are acceptable.
Do not mark Non-Executable merely because the report could be more detailed.
Do not require perfect precision, exact coordinates, exact quantities, launcher internals, account internals, network configuration, or internal cause unless that missing value is central to triggering the bug.

Executable means "actionable enough to attempt", not "confirmed reproducible".

============================================================
3. V16 ULTRA-NARROW BLOCKING-GATE FOR IMPROVED REPORTS
============================================================

Before finalizing an Executable label, check only for clearly blocking missing information.
This V16 gate is intentionally narrower than V15. It is designed to preserve the Executable cases that were incorrectly rejected by V15.

Do NOT use this gate as a general strictness rule. Do NOT reject a report merely because it could include more detail.
If the Target Improved Report already gives setup + action + observation and the missing information is only ordinary Minecraft/game knowledge, keep the label "Executable".

Mark the Target Improved Report as "Non-Executable" only when one of these blocking patterns clearly applies:

A. Unusual mob drop / command-block drop reports
- Non-Executable when the report claims an entity drops an impossible or unusual item, such as a command block, but does not specify the required custom setup, loot-table/data-pack/NBT/command configuration, or exact environment that makes the scenario attemptable.
- Do not assume custom loot or command setup if it is not stated.

B. Performance/lag reports involving trapdoors, stairs, or structures
- Non-Executable when the report only says to build/find a generic trapdoor/stair/block structure and observe lag, but omits the scale, density, layout, interaction, entity count, redstone state, or trigger that makes the lag reproduction attempt concrete.
- Executable when it gives a sufficiently constrained setup and action.

C. Jukebox / "Tears" music disc / audio playback state reports — ultra-narrow blocker
- Non-Executable when the issue is specifically about a jukebox or a named music disc such as "Tears" and the tester must evaluate silence, stopping, continuation, transition, or audio playback state, but the Target Improved Report does not explicitly establish the playback state being tested.
- To be Executable, the report must clearly provide: the audio source setup, the action/change being tested, and the expected or observed audio result.
- Do not infer that music is already playing just because a jukebox or music disc is named.
- This blocker is only for jukebox/music-disc playback-state reports. Do not apply it to unrelated rendering, skin, network, entity, crash, or movement reports.

D. Rail junction / redstone block / power-state layout reports — ultra-narrow blocker
- Non-Executable when the issue specifically depends on a rail junction, redstone block, power state, rail update, orientation, or activation order, but the Target Improved Report does not provide enough layout/power-state information to build one intended setup.
- A generic instruction such as placing rails/redstone blocks and observing a result is not enough when the bug depends on the exact rail junction layout, orientation, or power update.
- Executable when the report states one concrete configuration and action sequence that a tester can build without inventing the layout.
- This blocker is only for rail/redstone/power-state layout reports. Do not apply it to copper golems, dragon hitboxes, Discord streaming, skin loading, multiplayer pause, authentication, or crash/exception reports.

E. Visual rendering, glow, lava/fire, animation, and item-state reports — preserve Executable unless truly blocked
- These reports are NOT automatically Non-Executable.
- Keep them Executable when the report names the item/block/entity/environment, gives an action or state to create, and states the visual or animation outcome to observe.
- Mark them Non-Executable only if the central item/entity/environment or the key action needed to create the state is missing entirely.

If a blocking pattern applies, use label "Non-Executable" and failure_type "Missing Information" unless the provided information is mechanically impossible, in which case use "Wrong Information".
If no blocking pattern clearly applies and the report gives setup + action + observation, use label "Executable".

============================================================
4. V16 EXECUTABLE-PRESERVING CALIBRATION
============================================================

The following cases should usually remain "Executable" when the Target Improved Report provides a concrete setup/action plus an observable result. Do not reject them just because more implementation detail would be helpful:

- HTTP/HTTPS, skin loading, launcher/web/resource loading, skin/cape visibility, profile/device-change cases. Do not require exact network settings, launcher configuration, or account-linking details unless the report explicitly makes them the central trigger.
- Phantom spawn/no-sleep cases. Do not reject merely because the report does not restate the exact three-day Minecraft rule if the path to stay awake/trigger phantom spawning is attemptable by general game knowledge.
- Piston/block movement cases. If the report states the pushed/moved block action, breaking action, or interaction with the moving block, do not require frame-perfect timing unless the report gives no concrete sequence at all.
- Graphical freeze/rendering freeze cases. If the report gives a named state/action and the observable freeze/result, do not require the internal graphics state or root cause.
- Discord streaming/share-screen cases. If the streaming/share action and affected Minecraft behavior are stated, do not require Discord settings unless the report says the bug depends on a specific Discord setting.
- Sprint/double-tap, elytra/firework, dragon hitbox/projectile, copper golem/statue/oxidation, raid/vindicator spawn, item swap, pause/multiplayer, mining/placing, item position/height accessibility, and crash/exception cases remain Executable when setup + action + observation are present.

These calibration rules do not override the YAML. They mean: if the missing information is not a blocking precondition, keep the report Executable.

The Expected Behavior is unclearly defined or does not align with the game environment, mechanics, or character features.

Mapping:
- expected_behavior.presence = "Present" if the Target Improved Report explicitly states the desired/correct outcome.
- expected_behavior.presence = "Not Present" if the expected outcome is absent or only inferred from Observed Behavior.
- expected_behavior.quality = "Accurate" if clearly defined and aligned with Minecraft mechanics/environment.
- expected_behavior.quality = "Inaccurate" if unclear or conflicting with Minecraft mechanics/environment.
- expected_behavior.quality = "Not Applicable" if presence is "Not Present".

Expected Behavior must not control the S2R label by itself.
A report can have clear Expected Behavior but Non-Executable S2R if the reproduction path is blocked.

============================================================

observed_behavior.present.description:

Mapping:
- observed_behavior.presence = "Present" if the Target Improved Report describes what actually happened or how the environment responded.
- observed_behavior.presence = "Not Present" if no actual behavior/environment response is described.
- observed_behavior.quality = "Sufficient" if the specific outcome is clear enough to understand.
- observed_behavior.quality = "Insufficient" if incomplete, unclear, misleading, or assumption-heavy.
- observed_behavior.quality = "Not Applicable" if presence is "Not Present".

Observed Behavior must not control the S2R label by itself.
A report can have clear Observed Behavior but Non-Executable S2R if the reproduction path is blocked.
